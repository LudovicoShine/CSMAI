# 场景表 Schema（10 字段）

每个主场景与补充场景都必须按下表 10 个字段输出。

## 字段定义

| 项目 | 内容要求 |
|---|---|
| **场景标题** | 一眼看懂在解决什么问题，不要泛词（例：「经营总览驾驶舱」「项目推进与异常闭环看板」） |
| **场景描述** | 必须写清 4 件事：① 背景（客户当前在什么状态）② 价值（解决了什么管理动作）③ 为什么值得做（业务必要性）④ 为什么现在做（时机匹配） |
| **面向对象** | 具体到角色，例：「总经理、分管领导、运营负责人、向高层汇报的经营管理角色」 |
| **业务动向** | 这个场景出现在客户的哪些会议 / 动作里，例：「月度经营会、周经营例会、专题复盘会」 |
| **分析思路** | 从总览到下钻的判断链，例：「先看整体经营态势 → 再按部门/项目/客户拆解 → 再识别异常 → 再判断优先干预对象」 |
| **分析指标** | 列出指标名，必要时一句话补充含义，避免只列名称 |
| **与 BI / FDL / 简道云 / Dora 的相关性** | 必须融合：① BI 目标 ② 页面核心结构（顶部/中部/底部） ③ 故事链（打开看板的判断路径） ④ FDL 重点（要统一哪些字段/口径） ⑤ 简道云/Dora 是否适用 |
| **场景引导** | 在现场如何引导用户聊出这个场景，至少 2-3 个连续追问的范例 |
| **风险与痛点** | 这个场景落地容易卡在哪里（数据、口径、组织、责任） |
| **切入点提示** | 当客户说出某些关键词时，CSM 可以立即抛出这个场景。例：「当客户提到'老板总追问''每次经营会都得先统一数字'时，可切入该场景」 |

## 模板（直接复用）

```markdown
### 场景 S{n}
| 项目 | 内容 |
|---|---|
| 场景标题 | ... |
| 场景描述 | 背景：... 价值：... 为什么值得做：... 为什么现在做：... |
| 面向对象 | ... |
| 业务动向 | ... |
| 分析思路 | 先 ... → 再 ... → 再 ... → 最后 ... |
| 分析指标 | 指标A（含义）、指标B（含义）、指标C ... |
| 与 BI / FDL / 简道云 / Dora 的相关性 | **BI 目标**：... ; **页面核心结构**：顶部 KPI / 中部 ... / 底部 ... ; **故事链**：... ; **FDL 重点**：... ; **简道云**：是否适用 ; **Dora**：是否适用 |
| 场景引导 | 先问"..."，再问"..."，最后追问"..." |
| 风险与痛点 | ... |
| 切入点提示 | 当客户提到"..."时，可切入此场景 |
```

## 常见错误

- ❌ 场景标题写成"经营分析"这种泛词 → 没有信息量
- ❌ 场景描述只写"这个看板帮老板看数据" → 没有时机判断
- ❌ 分析指标只列"达成率、完成率" → 没有含义就是字典
- ❌ 与产品的相关性只写"用 BI 做" → 没有结构、故事链、口径治理就没用
- ❌ 场景引导只写"问客户怎么看" → 没有具体追问示范
