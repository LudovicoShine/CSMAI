## 1. 客户与部门业务情况介绍和推测

### 1.1 客户公司业务介绍
基于客户名称、行业属性以及此类机构的公开信息特征判断，中建政研集团大概率属于**政策研究、咨询服务、项目型交付**并存的组织。这类公司的经营管理，通常不会只盯单一财务结果，而会同时关注：项目推进是否正常、成果交付是否及时、重点客户是否稳定、经营汇报是否顺畅、不同部门对同一经营数据是否口径一致。

因此，这类组织做 BI 的核心价值，往往不是“把已有数据做成看板”，而是帮助管理层与运营层回答三类问题：
1. 现在整体经营和项目推进到底跑得怎么样；
2. 哪些项目、客户、部门或节点出了问题；
3. 出问题之后，接下来该由谁推动、先处理什么。

### 1.2 拜访部门业务介绍
#### 运营部门
运营部门大概率承担项目推进统筹、经营材料汇总、异常事项跟踪、领导汇报支撑、跨部门协调等职责。它更关心的是：
- 重点项目现在到底进展如何；
- 哪些事项在卡住；
- 哪些内容需要向领导快速说明；
- 经营材料如何更快、更准地形成。

#### IT部门
IT 部门大概率承担 BI 平台维护、系统接口支撑、数据接入、字段解释、账号权限、公共数据口径维护等职责。它更关心的是：
- 数据到底来自哪里；
- 为什么同一个指标在不同地方不一致；
- 哪些字段或规则最容易引发业务争议；
- 如何把业务诉求稳定地落到系统与数据层。

### 1.3 本次拜访判断
本次拜访不是单纯讨论“BI 好不好用”，而是一个标准的**业务场景线 + 数据治理线双线并行**调研。

业务场景线要回答：
- 运营部门最真实、最高频、最值得优先做的场景是什么；
- 这些场景到底服务谁、支撑什么管理动作、最终要解决什么问题。

数据治理线要回答：
- 为什么 BI 已用了很久，但仍然存在一系列使用问题；
- 这些问题里，哪些是因为公共数据、字段、口径、来源没有统一造成的；
- 数据治理应该怎样和真实业务场景绑定推进，而不是空转。

### 1.4 已知信息
- 客户：中建政研集团
- 行业：政策研究咨询
- 拜访部门：运营部门 + IT部门
- 当前状态：BI 使用时间较长
- 当前问题：公共数据一直没有治理，现有 BI 使用中存在一系列问题
- 拜访目标：既要解决数据场景问题，又希望补充和深化运营部门的业务场景

### 1.5 合理推测
从当前已知信息看，这家公司最值得优先确认的，未必是“还能不能再做新看板”，而是：
- 运营部门真实在用的材料和真实在做的动作是什么；
- 高层真正关心的问题是什么；
- 现有 BI 为什么没有嵌入这些动作；
- 数据问题到底卡住了哪些关键场景。

因此，本次拜访最好的推进方式，不是空谈数据治理，也不是泛泛讲行业方案，而是：
> **先锁定 1-2 个真正高频且高价值的运营/经营场景，再判断公共数据治理要如何为这些场景服务。**

## 3. 总调研框架

### 3.1 本次调研总目标
本次调研要形成四个明确结论：
1. 运营部门当前最真实、最高频、最值得优先做的业务场景是什么；
2. 这些场景当前是如何被支持的，哪些地方靠系统，哪些地方靠人工；
3. 公共数据未治理究竟卡住了哪些具体业务动作；
4. 下一步更适合先做业务场景验证、先补数据底座，还是双线并行。

### 3.2 调研方法论
建议遵循：**先识别人，再识别动作；先识别真实材料，再识别判断逻辑；先识别业务问题，再识别数据问题；最后再谈方案与推进。**

### 3.3 六大调研维度
- 岗位认知层：这个人到底负责什么、服务谁、向谁汇报
- 汇报材料层：他平时到底在做哪些材料、哪些内容最常被追问
- 现状痛点层：哪些工作最耗时、最不准、最难协同
- 数据源层：哪些数据来自系统、哪些来自 Excel、哪些靠人工解释
- 总经理关注层：高层真正关心什么、怎么看经营好坏
- 行业实践参考层：如果做深化，哪些场景最值得延展

### 3.4 完整版现场引导问题框架

#### 1. 基础信息确认
> 目标：先把人、职责、汇报关系、使用频率问清楚，避免一上来就讨论功能。

1. 你当前岗位主要负责哪些事情？其中哪些和经营、项目、材料汇报直接相关？  
   - 调研目的：明确这个人是不是核心场景 owner  
   - 追问方向：哪些是自己负责，哪些只是配合；哪些事情最花时间  
   - 适用干系人：全部，尤其是部门负责人/执行人

2. 你平时主要服务谁、向谁汇报、和谁协同最多？  
   - 调研目的：判断场景面向对象和汇报链条  
   - 追问方向：高层/部门负责人/项目负责人/支持部门谁最常提要求  
   - 适用干系人：全部

3. 你日常看数、做材料、开会讨论的频率大概是怎样的？  
   - 调研目的：判断这个场景是高频刚需还是低频需求  
   - 追问方向：日、周、月、专题、临时汇报的差异  
   - 适用干系人：运营负责人、执行人、高层支持角色

4. 现在这些工作里，哪些是系统直接支持的，哪些还是靠人手工完成？  
   - 调研目的：区分系统问题与流程问题  
   - 追问方向：哪些动作已经有 BI，哪些虽然有 BI 但不用，为什么不用  
   - 适用干系人：运营、IT

#### 2. 现状与痛点确认
> 目标：把“现在最痛的事情”问透，不要只停留在“数据不准”“系统不好用”这种抽象表述。

1. 最近一次让你明显觉得麻烦、低效或解释不清的情况是什么？  
   - 调研目的：抓真实案例而不是抽象抱怨  
   - 追问方向：发生在什么会、什么材料、哪个项目/部门、谁提出问题  
   - 适用干系人：全部

2. 现在最难拿到的数据是什么？最容易打架的指标或字段是什么？  
   - 调研目的：识别数据治理切口  
   - 追问方向：来源系统、责任人、口径冲突场景  
   - 适用干系人：IT、运营、材料整理人

3. 在你看来，现有 BI 最大的问题是“没有”“不准”“不好用”还是“用不上”？  
   - 调研目的：判断问题类型  
   - 追问方向：是场景不对，还是数据底座不对，还是呈现方式不对  
   - 适用干系人：全部

4. 如果只解决一个最痛的问题，你最希望先解决哪一个？  
   - 调研目的：帮助收敛优先级  
   - 追问方向：为什么它最痛，不解决会影响什么管理动作  
   - 适用干系人：部门负责人、高层、运营负责人

#### 3. 场景与指标确认
> 目标：把真实业务场景、判断逻辑和指标体系问清楚，避免只停留在“想看一个看板”。

1. 你平时最常做的三类经营/运营材料分别是什么？分别给谁看？  
   - 调研目的：锁定真实高频场景  
   - 追问方向：哪类材料每次都要做、哪类最费时间、哪类最常被追问  
   - 适用干系人：运营负责人、执行人

2. 老板/领导最常问的三个问题是什么？你通常怎么回答？  
   - 调研目的：锁定高层场景和判断逻辑  
   - 追问方向：哪些问题现在回答得慢、回答得不准、回答时需要拉很多人  
   - 适用干系人：高层支持角色、运营负责人、部门负责人

3. 对于“做得好/做得不好”，你们内部通常是靠哪些指标或判断规则来定的？  
   - 调研目的：识别指标体系和业务经验  
   - 追问方向：是固定规则、经验判断，还是领导口头标准；这些标准是否稳定  
   - 适用干系人：部门负责人、运营负责人、高层

4. 同一个指标如果不同人说法不一样，最后谁说了算？  
   - 调研目的：识别指标 owner 和治理责任人  
   - 追问方向：有没有正式口径文档；IT 是执行者还是解释者  
   - 适用干系人：IT、运营负责人、管理者

5. 如果先做一个看板，你最希望它帮助你减少什么动作、提升什么判断？  
   - 调研目的：把 BI 建设目标落到管理动作上  
   - 追问方向：减少拼材料、提前预警、统一口径、提升汇报效率、支撑专题分析等  
   - 适用干系人：全部

#### 4. 推进与协同确认
> 目标：把后续怎么推、谁拍板、谁配合、谁会卡住问清楚，做好干系人管理。

1. 如果我们这次先落一个 demo，最值得先做哪一个场景？  
   - 调研目的：形成首个推进抓手  
   - 追问方向：为什么是它；是不是大家都认可；有没有争议  
   - 适用干系人：部门负责人、运营负责人、高层支持角色

2. 这个场景如果推进，需要谁拍板、谁提供数据、谁负责确认口径？  
   - 调研目的：识别关键干系人和推进闭环  
   - 追问方向：谁会支持、谁会犹豫、谁最可能成为阻力  
   - 适用干系人：运营、IT、管理者

3. 如果推进过程中出现口径冲突、部门配合不到位、数据缺失，通常谁来协调？  
   - 调研目的：判断组织协同成熟度  
   - 追问方向：过去类似问题是谁解决的，用什么机制解决  
   - 适用干系人：运营负责人、管理者、IT接口人

4. 你希望下次沟通时看到什么成果，才会觉得这次讨论是有价值的？  
   - 调研目的：明确下一步输出物标准  
   - 追问方向：是 demo、是指标清单、是数据治理清单、还是推进路线图  
   - 适用干系人：全部

### 3.5 干系人调研判断提醒
- **高层/高层支持角色**：重点问“关注什么、怎么判断、要什么结果”，不要陷入字段细节。
- **部门负责人**：重点问“哪些动作最需要支持、哪些场景最值得先做、哪些问题最影响管理效率”。
- **执行人/材料整理人**：重点问“具体怎么做、哪里最花时间、哪一步最容易出错、哪些材料最难整理”。
- **IT/数据负责人**：重点问“数据从哪来、哪个字段最容易冲突、谁负责定义、系统边界在哪里”。

### 场景 S2
| 项目 | 内容 |
|---|---|
| 场景标题 | 项目推进与异常闭环看板 |
| 场景描述 | 这是一个面向运营部门高频使用的项目管理场景，核心目标是让运营负责人在周会或日常推进中，看清重点项目当前推进到哪一步、哪些节点已超期、哪些问题尚未闭环、哪些项目需要优先协调资源。这个场景的价值在于，它直接连接运营部门“真正会用”的动作，而不是停留在抽象的数据治理上。对于中建政研这类可能具备项目型交付特征的组织来说，项目推进的透明度决定了经营协同效率，也最容易成为 BI 深化的第一批真实落地场景。 |
| 面向对象 | 运营负责人、项目统筹角色、项目管理支持人员 |
| 业务动向 | 周会、项目推进会、异常协调会、重点事项跟踪会 |
| 分析思路 | 先看整体推进是否正常，再看红黄绿项目，再定位异常节点和责任人，再判断资源和动作 |
| 分析指标 | 节点完成率、逾期节点数、问题闭环率、重点项目进度、协同响应时长 |
| 与 BI / FDL 的相关性 | **BI 目标**：支撑运营部门高频查看项目推进状态；**页面核心结构**：项目列表、节点进度条、异常预警区、责任分布区；**故事链**：先看本周项目整体状态 → 再看红灯项目 → 下钻到具体节点和责任 → 形成催办动作；**FDL 重点**：统一项目编号、节点定义、异常状态、责任角色映射。 |
| 场景引导 | 先问：“你们周会最常盯哪些项目？”再问：“项目状态不一致时，通常是谁来拍板？”最后追问：“哪些问题最容易拖到下周还没闭环？” |
| 风险与痛点 | 项目节点标准不统一、不同项目负责人汇报口径不同、异常定义模糊，导致 BI 即使有数据也难真正支撑管理动作。 |
| 切入点提示 | 当客户提到“周会就盯进度”“项目状态经常说不清”“异常问题总是靠群里追”时，可切入此场景。 |

### 场景 S3
| 项目 | 内容 |
|---|---|
| 场景标题 | 经营材料自动化汇总场景 |
| 场景描述 | 该场景聚焦于“汇报材料怎么来”的问题。许多组织并不是没有数据，而是每周、每月、每次专题汇报前，都要花大量时间把不同表、不同系统、不同口径的数据重新拼成材料。这个场景的核心价值，是帮助运营部门减少重复性整理劳动，把时间从“拼材料”转回“看问题、做判断”。如果中建政研当前确实存在“每次材料都要人肉整理”的情况，那这是非常典型且高价值的 BI 深化切口。 |
| 面向对象 | 运营部门材料整理人、汇报负责人、部门管理者 |
| 业务动向 | 周报、月报、专题汇报、经营分析材料准备 |
| 分析思路 | 先拆材料结构，再拆来源数据，再识别哪些能标准化，最后形成半自动/自动生成逻辑 |
| 分析指标 | 材料准备时长、重复取数次数、人工修订次数、口径冲突次数、汇报等待时长 |
| 与 BI / FDL 的相关性 | **BI 目标**：将常规经营材料所需数据标准化沉淀到看板与主题分析中；**页面核心结构**：汇总 KPI、专题分析块、附录明细区；**故事链**：先看总览 → 再导出/复用材料核心页 → 补充少量文字判断；**FDL 重点**：统一材料字段来源、统计周期、组织口径和导出规则。 |
| 场景引导 | 问：“最近一次汇报材料做了多久？”“哪些内容每次都得重新找？”“如果能自动生成 70%，你最希望先自动哪部分？” |
| 风险与痛点 | 即使有 BI，如果输出结构不对、字段不统一、导出逻辑不稳定，最后还是会回到 Excel 和 PPT 手工加工。 |
| 切入点提示 | 当客户提到“材料总是要重新拼”“每次汇报前都很赶”“很多数要反复确认”时，可切入此场景。 |

### 场景 S4
| 项目 | 内容 |
|---|---|
| 场景标题 | 公共数据口径与字段治理场景 |
| 场景描述 | 这是本次拜访中最明确的一条数据治理线。它不是为了单独做一个治理项目而做，而是为了支撑前面那些真实业务场景能“看得准、讲得通、用得下去”。如果公共字段、组织层级、项目状态、客户分类这些底层概念长期没有统一，那么无论做多少看板，最终都容易陷入“展示很多、决策很难”的状态。这个场景值得推进，但不应孤立推进，而应与高频业务场景绑定。 |
| 面向对象 | IT 负责人、数据管理角色、运营对接人、关键业务口径 owner |
| 业务动向 | 数据治理例会、字段口径对齐会、系统接入评审、指标定义确认 |
| 分析思路 | 先识别争议字段，再看来源系统，再统一定义，再形成规则，再绑定业务场景验证 |
| 分析指标 | 字段覆盖率、口径一致率、数据完整率、冲突字段数、标准落地率 |
| 与 BI / FDL 的相关性 | **BI 目标**：支撑所有高频业务看板的稳定输出；**页面核心结构**：字段字典、来源映射、口径差异说明、治理进度；**故事链**：先识别争议字段 → 再追溯来源 → 确认 owner → 建规则 → 绑定业务验证；**FDL 重点**：字段命名、映射规则、主题层定义、指标依赖关系。 |
| 场景引导 | 先问：“现在最容易打架的字段是什么？”再问：“这些字段最后谁拍板定义？”最后问：“如果先只治理 10 个字段，你们会选哪 10 个？” |
| 风险与痛点 | 治理如果脱离业务场景，会变成空泛工程；如果责任人不清，统一规则很难落地。 |
| 切入点提示 | 当客户明确提到“公共数据没治理”“字段不统一”“不同系统对不上”时，可切入此场景。 |

### 场景 S5
| 项目 | 内容 |
|---|---|
| 场景标题 | 重点客户/重点项目专题经营分析 |
| 场景描述 | 除了日常总览和推进，很多客户还会在某些关键时点对重点项目、重点客户、重点专题做更深的经营分析。这个场景的价值在于，它能把 BI 从“固定展示”进一步升级成“管理决策支持”。如果中建政研存在重要客户项目或战略专题，那么此类专题分析场景很适合作为第二阶段深化方向。 |
| 面向对象 | 高层、运营负责人、专题负责人 |
| 业务动向 | 专题会、重点客户复盘、重点项目分析会 |
| 分析思路 | 先看总体专题目标，再拆过程指标，再识别异常，再看资源和行动建议 |
| 分析指标 | 专题达成率、重点客户状态、关键节点完成率、资源投入产出比、异常分布 |
| 与 BI / FDL 的相关性 | **BI 目标**：支持“按专题/按重点对象”快速分析；**页面核心结构**：专题总览、过程拆解、异常明细、行动建议区；**故事链**：打开专题 → 看目标与现状偏差 → 找异常 → 拆原因 → 做动作；**FDL 重点**：重点客户标记、专题标签、项目归属和多维筛选能力。 |
| 场景引导 | 先问：“有没有哪些项目/客户是老板会单独拎出来问的？”再问：“这些专题现在怎么分析？” |
| 风险与痛点 | 专题类场景容易边界不清、需求变化快，如果没有稳定框架容易反复返工。 |
| 切入点提示 | 当客户提到“重点项目要单独看”“某个客户领导特别关心”时，可切入此场景。 |

### 场景 S6
| 项目 | 内容 |
|---|---|
| 场景标题 | 跨部门协同与责任闭环场景 |
| 场景描述 | 当经营问题真正落地时，往往不是“看到了就结束”，而是要跨部门协同去处理。这个场景聚焦于：某个异常出现后，谁负责认领、谁负责协调、多久闭环、是否形成复盘。它的价值在于把 BI 从“看问题”推进到“推动动作”。如果客户当前存在跨部门信息断层、问题跟踪分散、责任不清的情况，这会是一个非常重要的延展场景。 |
| 面向对象 | 运营协调角色、部门负责人、支持部门接口人 |
| 业务动向 | 异常协调会、跨部门问题跟进、周度复盘 |
| 分析思路 | 先识别异常 → 再认领责任 → 再追踪动作 → 再看闭环结果 |
| 分析指标 | 问题闭环率、平均闭环时长、重复问题数、跨部门响应时长、责任明确率 |
| 与 BI / FDL 的相关性 | **BI 目标**：让异常问题能被跟踪和复盘；**页面核心结构**：问题池、责任分布、处理进度、复盘记录；**故事链**：发现异常 → 指派责任 → 跟踪进度 → 复盘结果；**FDL 重点**：责任角色、部门映射、问题状态流转、复盘标签。 |
| 场景引导 | 问：“问题发现以后通常谁来跟？”“跨部门最容易卡在哪里？”“怎么判断一个问题算闭环？” |
| 风险与痛点 | 如果组织协同机制弱，即使看板很完整，也可能只能‘看见问题’，无法推动解决。 |
| 切入点提示 | 当客户提到“跨部门经常扯皮”“问题没人跟到底”“异常总是重复出现”时，可切入此场景。 |

## 7. 场景地图

### 7.1 业务域 × 角色层级地图
| 业务域 / 角色层级 | 高层管理者关注什么 | 部门负责人关注什么 | 执行/支持角色关注什么 | BI 可怎么切入 |
|---|---|---|---|---|
| 经营域 | 整体经营是否健康、重点项目是否偏离 | 经营动作怎么拆解、哪些问题最影响结果 | 材料如何形成、数据如何汇总 | 做经营总览驾驶舱、经营拆解分析页 |
| 项目域 | 哪些重点项目会影响整体结果 | 项目进度、异常、责任闭环 | 节点维护、问题跟踪、项目状态填报 | 做项目推进与异常闭环看板 |
| 数据域 | 为什么同一个数解释不一致 | 哪些字段/口径最影响管理动作 | 字段来源、映射、标准定义 | 做关键字段字典、来源映射与口径清单 |
| 协同域 | 关键问题有没有人推动 | 哪些事项跨部门最容易卡 | 谁来认领、谁来跟进、怎么闭环 | 做协同问题池与责任追踪页 |
| 风险域 | 哪些风险会放大到经营层面 | 哪些异常需要提前干预 | 异常如何识别、记录、处理 | 做异常预警与干预追踪页 |
| 汇报域 | 有没有一页就能看懂的材料 | 材料是否足够支撑管理判断 | 哪些内容每次都要重复拼 | 做一页式经营汇报页和材料自动化输出 |

### 7.2 BI 建设落点图
| 场景 | BI 目标 | 关键分析方式 | 当前优先级 | 对应业务域 |
|---|---|---|---|---|
| 经营总览驾驶舱 | 让高层 5 分钟判断整体态势 | 总览 + 下钻 + 异常识别 | 高 | 经营域 |
| 项目推进与异常闭环 | 让运营高频跟进项目状态 | 进度跟踪 + 红黄绿预警 + 闭环追踪 | 高 | 项目域 |
| 经营材料自动化汇总 | 减少手工拼材料 | 模板化汇总 + 导出复用 | 高 | 汇报域 |
| 公共数据口径与字段治理 | 支撑看数可信 | 字段映射 + 口径统一 + owner 管理 | 中高 | 数据域 |
| 跨部门协同与责任闭环 | 让问题真正推动起来 | 问题池 + 责任链 + 进度跟踪 | 中 | 协同域 |
| 经营异常预警 | 让问题从事后走向事中 | 阈值预警 + 分类干预 | 中 | 风险域 |

### 7.3 场景优先级建议
| 场景 | 当前适合度 | 原因 | 建议顺序 |
|---|---|---|---|
| 经营总览驾驶舱 | 高 | 对高层价值感知最强 | 先做 |
| 项目推进与异常闭环 | 高 | 对运营高频使用最直接 | 先做 |
| 经营材料自动化汇总 | 高 | 最容易立即减少人工劳动 | 先做 |
| 公共数据口径与字段治理 | 中高 | 是底座，但应绑定业务推进 | 第二阶段 |
| 跨部门协同与责任闭环 | 中 | 依赖组织协同机制 | 第二阶段 |
| 经营异常预警 | 中 | 需要先稳定基础口径 | 后续深化 |

> 建议现场优先围绕“经营总览 + 项目推进 + 经营材料自动化”三类场景展开，因为它们最容易同时验证业务价值与数据治理必要性。
