# CSM AI Skills 版本迭代记录

本文档用于对外展示 `Pre-Visit Framework` 与 `Post-Visit Landing` 两个 CSM AI Skills 的设计演进过程，记录从初始框架梳理到平台化适配的主要需求、迭代动作与业务价值。

## 项目背景

本项目围绕客户成功团队的拜访前准备与拜访后收敛两个核心工作流，设计并优化两套 AI Skills：

- `Pre-Visit Framework`：用于拜访前客户背景判断、业务流程梳理、场景设计、现场追问和推进建议。
- `Post-Visit Landing`：用于拜访后根据录音、纪要和 CSM 标注，沉淀业务经验、收敛真实场景、生成落地路径和客户汇报材料。

整体目标不是做通用问答助手，而是把 CSM 团队在客户拜访、场景挖掘、BI 落地判断中的隐性经验，沉淀为可复用、可培训、可调试、可持续迭代的 AI 工作流。

## V1.0 双 Skill 基础梳理

### 需求来源

最初本地已有两个由不同 AI 分别生成的 Skills，结构、术语和输出边界不完全统一，需要整理成更规范的 Skill 格式。

### 关键迭代

- 梳理 `Pre-Visit Framework` 与 `Post-Visit Landing` 的职责边界。
- 将两个 Skill 统一为“主 `SKILL.md` + 引用资料 + 模板/脚本”的结构。
- 精简主说明，把详细 schema、产品分层、访谈问题、实施路径下沉到 reference 文件。
- 统一前后访谈中的术语，如主场景、补充场景、业务经验、Quick Win、客户汇报材料等。

### 业务价值

- 让两个 Skill 从“可用 prompt”升级为“可维护方法论”。
- 明确拜访前负责“设计问题和场景”，拜访后负责“收敛证据和落地方案”。
- 为后续培训新人、复盘客户案例、调试模型输出打下结构基础。

## V1.1 Pre/Post 闭环示例建设

### 需求来源

`Post-Visit` 缺少端到端示例，而 `Pre-Visit` 已有“中建政研集团”访前示例。希望用同一客户补齐访后示例，形成完整闭环。

### 关键迭代

- 新增“中建政研集团”对应的 Post-Visit 端到端示例。
- 在访后示例中体现 `skill1_output` 的使用方式。
- 将访前预判与访后现场反馈做对照，包括：
  - 被验证的场景
  - 现场新增的场景
  - 未涉及的场景
  - 被否定或暂缓的场景
- 强化 P0/P1/P2 优先级、Quick Win MVP 和客户汇报材料结构。

### 业务价值

- 让 CSM 新人能看到一次完整的“访前准备 → 现场反馈 → 访后落地方案”闭环。
- 方便用同一客户案例做模型调试与输出质量对比。
- 验证 Pre-Visit 输出是否真的能被 Post-Visit 接住。

## V1.2 场景库与钩子词库沉淀

### 需求来源

早期 6 个主场景主要靠模型临时生成，缺少长期沉淀。现场“钩子词”也散落在正文中，不方便 CSM 反查。

### 关键迭代

- 新增“行业 × 部门”场景库，覆盖制造、零售、政务、咨询、医药等典型行业。
- 将运营、财务、HR、供应链、营销等部门的常见 BI 场景沉淀为可索引模板。
- 新增钩子关键词词库，形成“客户原话关键词 → 命中场景 → 建议追问”的查找表。
- 将“老板总追问”“材料每次都要重新拼”“口径打架”“项目状态说不清”等高频信号结构化。

### 业务价值

- 把 CSM 团队的隐性行业经验显性化。
- 拜访某行业某部门时，AI 能优先召回成熟模板，而不是从零生成。
- CSM 现场可以根据客户原话快速判断切入场景和追问方向。

## V1.3 质量自检与交付模板增强

### 需求来源

输出约束原本散落在 `SKILL.md` 里，模型生成后缺少稳定自检机制；同时 CSM 经常需要打印、转 PDF 或复制到飞书/钉钉。

### 关键迭代

- 为两个 Skill 分别新增 `quality_checklist.md`。
- `Pre-Visit` 自检覆盖信息完整性、6 个主场景、场景 schema、钩子雷达、推进建议和场景地图。
- `Post-Visit` 自检覆盖 transcript 分级、业务经验数量、原文依据、场景收敛、Quick Win、客户汇报材料和碎片化处理。
- 为 HTML 输出模板增加打印友好样式，支持白底黑字、A4 分页、表格续页。
- 新增 Markdown-only 模板，方便复制到飞书、钉钉或邮件正文。

### 业务价值

- 让模型输出从“生成完成”变成“生成后自检再交付”。
- 提高不同模型、不同使用者之间的质量稳定性。
- 支持 CSM 在 Web、PDF、文档协同工具之间灵活交付。

## V1.4 客户简报子产物

### 需求来源

Post-Visit 第 7 节“客户汇报材料”原本嵌在内部大文档中，容易混入内部判断、风险、置信度等不适合发给客户的信息。

### 关键迭代

- 新增独立客户版简报模板。
- 将客户可见内容从内部 CSM 文档中切片出来。
- 明确客户版只保留：
  - 一句话总结
  - 推荐场景一页纸
  - 预期价值
  - 建议交付时间
  - 需要客户配合事项
  - 下一步安排
- 明确禁止暴露业务经验原文、置信度、当前阻塞、内部风险判断、否定标签和 CSM 内部备注。

### 业务价值

- 避免内部材料直接外发导致信息泄漏。
- 让 CSM 能一键生成“只给客户看的版本”。
- 提升客户沟通材料的专业度和安全边界。

## V1.5 Post-Visit 轻量自动化

### 需求来源

希望从“录音/纪要 → Post-Visit Skill 输入”之间建立一条轻量流水线，为后续接 Whisper 或 Dora 做准备。

### 关键迭代

- 新增 `prepare_post_visit.py` 脚本。
- 支持读取 transcript、CSM annotations、Pre-Visit 输出、会议日期、客户行业等输入。
- 自动根据文本长度和模糊标记判断 `transcript_quality`：
  - 完整清晰
  - 部分模糊
  - 碎片化
- 输出标准化 JSON payload，可直接喂给 Post-Visit Skill。
- 预留 OpenAI Whisper / Dora Provider，但本阶段不做真实外部接口接入。

### 业务价值

- 将访后材料输入标准化，减少手工整理成本。
- 让录音质量判断有基础规则可依赖。
- 为未来“录音 → 转写 → 访后方案”一键流水线打基础。

## V1.6 公司平台适配与示例移除

### 需求来源

公司平台级 Agent 产品使用 GLM、DeepSeek 等模型时，带 example 示例后质量下滑明显；同时平台只支持 `SKILL.md`、`SCRIPTS/`、`REFERENCES/` 三类结构。

### 关键迭代

- 移除 examples 示例入口，避免弱模型过度模仿示例。
- 清理 README 和 assets 入口，将模板统一迁移到 `REFERENCES/templates/`。
- 将脚本统一迁移到 `SCRIPTS/`。
- 更新所有路径引用为公司平台兼容结构。
- 在两个 Skill 中加入平台适配说明。

### 业务价值

- 降低 GLM、DeepSeek 等模型被示例牵引跑偏的概率。
- 提升 Skill 在公司平台中的可识别性和可部署性。
- 让 Skill 结构更贴合内部 Agent 产品规范。

## V1.7 弱模型友好化与 Handoff 机制

### 需求来源

在模型能力一般或上下文长度有限时，一次性生成完整大文档会导致输出广度和深度下降。用户反馈“拆开几个问题问，效果更好”。

### 关键迭代

- 为 Pre-Visit 增加分段 handoff 策略：
  - 先输出客户业务流程、部门工作流程、拜访判断
  - 再输出 6 个主场景
  - 再输出现场怎么聊、数据流转、系统边界、落地复杂度
  - 最后输出推进建议、场景地图和自检结果
- 为 Post-Visit 增加分段 handoff 策略：
  - 先输出拜访摘要、录音质量和业务经验
  - 再输出场景收敛
  - 再输出指标路径和看板建议
  - 最后输出推进建议、客户汇报和自检结果
- 要求每段结尾输出 handoff 摘要，说明已完成内容、仍缺什么、下一段继续什么。

### 业务价值

- 让弱模型也能稳定输出有深度的内容。
- 减少长文档一次性生成导致的遗漏。
- 方便 CSM 把复杂任务拆成多轮高质量协作。

## V1.8 白话化业务理解与访谈追问增强

### 需求来源

用户反馈当前输出仍偏抽象，需要“讲给完全不了解业务的人听”，并且要把公司、部门、岗位、日常工作、痛点和解决方案讲细。

### 关键迭代

- 在 Pre-Visit 中新增公司主要业务流程和部门业务流程要求。
- 明确要求用最白话、最容易理解的方式讲述：
  - 公司日常怎么运转
  - 对接部门每天/每周做什么
  - 岗位和部门之间如何协作
  - 痛点为什么发生
  - 帆软具体解决哪一步
- 强化场景表要求，必须写清：
  - 日常工作拆解
  - 痛点强度
  - 影响对象
  - 帆软解决方案
  - 现场继续聊的路径
- 补充高频访谈问题：
  - “您每天/每周花时间最多的事是什么？”
  - “这件事一般花多长时间？”
  - “从拿到数据到交出去，中间经过几步？”
  - “哪一步最头疼？”
  - “做慢或出错会影响谁？”

### 业务价值

- 让输出更适合 CSM 新人理解客户业务。
- 帮助 CSM 从“聊产品”转向“聊日常工作和真实痛点”。
- 更容易识别高频、高痛、高价值的 BI 落地场景。

## V1.9 反馈闭环机制

### 需求来源

需要在 Skill 内预留一个文件，持续记录大家使用过程中的反馈；如果平台支持 agent 反写，未来可以把反馈自动合并回 Skill。

### 关键迭代

- 为两个 Skill 分别新增 `feedback_log.md`。
- 将本轮所有问题反馈以“日期、来源、问题现象、影响、已采取动作、状态”的形式记录。
- 新增 `append_feedback.py` 脚本，支持把新反馈追加到反馈记录文件。
- 在 `SKILL.md` 中挂载反馈记录，要求后续迭代先读取反馈日志。

### 业务价值

- 让使用反馈不再散落在聊天记录里。
- 形成“使用 → 反馈 → 记录 → 反写 Skill”的持续优化机制。
- 为后续多人共创、平台级 Agent 运营和模型效果评估提供沉淀载体。

## 当前版本能力总结

当前两个 Skills 已从单次 prompt 工具演进为一套面向 CSM 拜访全流程的 AI 工作流：

| 阶段 | Skill | 核心能力 |
|---|---|---|
| 拜访前 | Pre-Visit Framework | 客户业务理解、部门流程梳理、场景生成、访谈问题、钩子词、推进建议 |
| 拜访后 | Post-Visit Landing | 录音/纪要收敛、业务经验提炼、场景优先级、指标路径、看板建议、客户汇报 |
| 质量保障 | 两个 Skill 共用机制 | 自检清单、弱模型 handoff、平台结构适配、反馈日志 |
| 交付支持 | 模板与脚本 | 打印友好模板、Markdown 模板、客户简报模板、访后预处理脚本 |

## 后续建议

- 用 3-5 个真实客户案例测试 GLM 和 DeepSeek 的输出稳定性。
- 对比“带示例”和“无示例”两种模式下的输出质量，确认示例移除后的收益。
- 将 `feedback_log.md` 纳入团队日常复盘，每周或每两周集中反写一次 Skill。
- 如果平台支持自动工具调用，可让 Agent 在每次使用后自动追加反馈或生成质量评分。
- 若平台不支持 `REFERENCES/templates/` 二级目录，可进一步把模板文件上移到 `REFERENCES/` 根目录。
