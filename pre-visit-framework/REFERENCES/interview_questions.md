# 完整版现场引导问题框架

按 **基础信息确认 → 现状与痛点确认 → 场景与指标确认 → 推进与协同确认** 四阶段组织。每个问题必须写清：调研目的 / 追问方向 / 适用干系人。

---

## 输入补齐提醒（访前问卷）

> 本节定义拜访前信息不足时的补齐问卷规范。完整交互规则、字段分类、固定选项列表见 SKILL.md「补齐问卷交互规范」章节。

### 核心原则

1. **选择题必填、填空题选填**：所有 `multiple_choice` / `single_choice` 题型为必填；所有 `text` 题型为非必填，用户留空时必须能正常提交。
2. **多选优先**：除 `customer_scale` 为单选外，其余选择题均为多选，且支持用户自由补充选项。
3. **交互方式**：优先使用 `ask_user_question` 工具弹出卡片；降级时用 Markdown 列表输出，并在每个填空题后标注"（选填，不填默认空白）"。
4. **分批展示**：一次最多 5 个问题，第一批必须包含 `visit_goal`、`visit_target_role`、`customer_bi_status`。

## 阶段 1 · 基础信息确认

> **目标**：先把人、职责、汇报关系、使用频率问清楚，避免一上来就讨论功能。

1. **你当前岗位主要负责哪些事情？其中哪些和经营、项目、材料汇报直接相关？**
   - 调研目的：明确这个人是不是核心场景 owner
   - 追问方向：哪些是自己负责，哪些只是配合；哪些事情最花时间
   - 适用干系人：全部，尤其是部门负责人/执行人

2. **你平时主要服务谁、向谁汇报、和谁协同最多？**
   - 调研目的：判断场景面向对象和汇报链条
   - 追问方向：高层/部门负责人/项目负责人/支持部门谁最常提要求
   - 适用干系人：全部

3. **你日常看数、做材料、开会讨论的频率大概是怎样的？**
   - 调研目的：判断这个场景是高频刚需还是低频需求
   - 追问方向：日、周、月、专题、临时汇报的差异
   - 适用干系人：运营负责人、执行人、高层支持角色

4. **现在这些工作里，哪些是系统直接支持的，哪些还是靠人手工完成？**
   - 调研目的：区分系统问题与流程问题
   - 追问方向：哪些动作已经有 BI，哪些虽然有 BI 但不用，为什么不用
   - 适用干系人：运营、IT

---

## 阶段 2 · 现状与痛点确认

> **目标**：把"现在最痛的事情"问透，不要只停留在"数据不准""系统不好用"这种抽象表述。

1. **最近一次让你明显觉得麻烦、低效或解释不清的情况是什么？**
   - 调研目的：抓真实案例而不是抽象抱怨
   - 追问方向：发生在什么会、什么材料、哪个项目/部门、谁提出问题
   - 适用干系人：全部

2. **现在最难拿到的数据是什么？最容易打架的指标或字段是什么？**
   - 调研目的：识别数据治理切口
   - 追问方向：来源系统、责任人、口径冲突场景
   - 适用干系人：IT、运营、材料整理人

3. **在你看来，现有 BI 最大的问题是"没有""不准""不好用"还是"用不上"？**
   - 调研目的：判断问题类型
   - 追问方向：是场景不对，还是数据底座不对，还是呈现方式不对
   - 适用干系人：全部

4. **如果只解决一个最痛的问题，你最希望先解决哪一个？**
   - 调研目的：帮助收敛优先级
   - 追问方向：为什么它最痛，不解决会影响什么管理动作
   - 适用干系人：部门负责人、高层、运营负责人

---

## 阶段 3 · 场景与指标确认

> **目标**：把真实业务场景、判断逻辑和指标体系问清楚，避免只停留在"想看一个看板"。

1. **你平时最常做的三类经营/运营材料分别是什么？分别给谁看？**
   - 调研目的：锁定真实高频场景
   - 追问方向：哪类材料每次都要做、哪类最费时间、哪类最常被追问
   - 适用干系人：运营负责人、执行人

2. **老板/领导最常问的三个问题是什么？你通常怎么回答？**
   - 调研目的：锁定高层场景和判断逻辑
   - 追问方向：哪些问题现在回答得慢、回答得不准、回答时需要拉很多人
   - 适用干系人：高层支持角色、运营负责人、部门负责人

3. **对于"做得好/做得不好"，你们内部通常是靠哪些指标或判断规则来定的？**
   - 调研目的：识别指标体系和业务经验
   - 追问方向：是固定规则、经验判断，还是领导口头标准；这些标准是否稳定
   - 适用干系人：部门负责人、运营负责人、高层

4. **同一个指标如果不同人说法不一样，最后谁说了算？**
   - 调研目的：识别指标 owner 和治理责任人
   - 追问方向：有没有正式口径文档；IT 是执行者还是解释者
   - 适用干系人：IT、运营负责人、管理者

5. **如果先做一个看板，你最希望它帮助你减少什么动作、提升什么判断？**
   - 调研目的：把 BI 建设目标落到管理动作上
   - 追问方向：减少拼材料、提前预警、统一口径、提升汇报效率、支撑专题分析等
   - 适用干系人：全部

---

## 阶段 4 · 推进与协同确认

> **目标**：把后续怎么推、谁拍板、谁配合、谁会卡住问清楚，做好干系人管理。

1. **如果我们这次先落一个 demo，最值得先做哪一个场景？**
   - 调研目的：形成首个推进抓手
   - 追问方向：为什么是它；是不是大家都认可；有没有争议
   - 适用干系人：部门负责人、运营负责人、高层支持角色

2. **这个场景如果推进，需要谁拍板、谁提供数据、谁负责确认口径？**
   - 调研目的：识别关键干系人和推进闭环
   - 追问方向：谁会支持、谁会犹豫、谁最可能成为阻力
   - 适用干系人：运营、IT、管理者

3. **如果推进过程中出现口径冲突、部门配合不到位、数据缺失，通常谁来协调？**
   - 调研目的：判断组织协同成熟度
   - 追问方向：过去类似问题是谁解决的，用什么机制解决
   - 适用干系人：运营负责人、管理者、IT接口人

4. **你希望下次沟通时看到什么成果，才会觉得这次讨论是有价值的？**
   - 调研目的：明确下一步输出物标准
   - 追问方向：是 demo、是指标清单、是数据治理清单、还是推进路线图
   - 适用干系人：全部

---

## 干系人调研判断提醒

| 干系人类型 | 该重点问什么 | 该避免什么 |
|---|---|---|
| **高层 / 高层支持角色** | 关注什么、怎么判断、要什么结果 | 不要陷入字段细节 |
| **部门负责人** | 哪些动作最需要支持、哪些场景最值得先做、哪些问题最影响管理效率 | 不要只问"用不用 BI" |
| **执行人 / 材料整理人** | 具体怎么做、哪里最花时间、哪一步最容易出错、哪些材料最难整理 | 不要谈战略 |
| **IT / 数据负责人** | 数据从哪来、哪个字段最容易冲突、谁负责定义、系统边界在哪里 | 不要只问技术问题，要追业务影响 |

---

## 调研方法论一句话

> **先识别人，再识别动作；先识别真实材料，再识别判断逻辑；先识别业务问题，再识别数据问题；最后再谈方案与推进。**
